**iconify_flutter**

Refined Production Plan

_v2.0 - March 12, 2026_

# **1\. Executive Summary**

This document is a ground-up refinement of the original production plan, incorporating new research findings that materially change several architectural decisions. Read the sections marked CRITICAL before writing any code.

WHAT CHANGED FROM v1: Seven new findings affect the plan significantly. The 4-week timeline is fiction. The

default hybrid-auto mode has an ethical problem. The flutter_svg Impeller risk was completely missing. The

data acquisition strategy for the CLI was wrong. Icon license complexity was understated. The starter registry

was undefined. build_runner was never properly evaluated vs the CLI-only approach.

# **2\. Critical Research Findings (Read First)**

These findings were absent from the original plan and change real architectural decisions.

## **2.1 - The Iconify Public API Is Not Free Infrastructure**

CRITICAL: Do NOT default to hitting the public Iconify API in production apps.

Research finding: Iconify themselves stated in their 2024 updates that API infrastructure costs are

significant and NOT covered by GitHub sponsorships. They explicitly said: 'It is better to encourage

use of alternative solutions that do not rely on API.' The Nuxt Icon module switched away from defaulting

to the public API for exactly this reason.

If iconify_flutter becomes popular and defaults to hitting api.iconify.design, you will be responsible

for flooding a free community-funded service with mobile app traffic. This is both an ethical problem

and a reliability risk - the API has no SLA and no rate limit enforcement.

REQUIRED CHANGE: The out-of-box strategy document's 'Hybrid Auto Mode' default must be revised:

- In debug/dev builds: remote fetch allowed (reasonable for development)
- In release/profile builds: remote fetch DISABLED by default - must be explicitly opted in
- Starter registry ships enough common icons so first-run experience is not blank
- CLI generate must be the strongly recommended production path

## **2.2 - The flutter_svg Impeller Risk Is Real and Unaddressed**

CRITICAL: flutter_svg has documented, unfixed rendering bugs with Impeller that directly affect icon use cases.

Confirmed issues (2024-2025):

• colorFilter applied to SVGs renders incorrectly with Impeller (Android + iOS)

• Linear gradients don't render correctly on iOS with Impeller

• SVGs render pixelated on Android emulators with Impeller

• The workaround (Transform.scale wrapper) is a hack, not a fix

Additional risk: flutter_svg's original author Dan Field died in 2024. The package has been forked to a

new community-maintained repo. This is a dependency you cannot control and cannot guarantee.

This is critical because: color theming and dynamic color overrides are core to your package's value

proposition. If colorFilter is broken with Impeller, your color: parameter on IconifyIcon is broken on

a large percentage of production devices.

REQUIRED ADDITIONS to the plan:

- Section 11 (Correctness Plan) must add: Impeller golden tests on real devices, not just emulators
- Rendering strategy must include a fallback path: when colorFilter is needed and Impeller is active, use the rasterization approach (render to ui.Image at target DPR) instead of SvgPicture directly
- Track flutter_svg's new community repo - add it as a listed dependency risk in governance docs
- Consider: for the generated/offline mode, pre-rasterize icons at build time as a future optimization

## **2.3 - CLI Data Source Must NOT Be the HTTP API**

The original plan implies syncing icon data from the Iconify HTTP API. This is wrong for the CLI sync step.

CORRECT approach: The Iconify icon-sets repository publishes all icon data as structured JSON in a GitHub

repository (iconify/icon-sets). This is the canonical data source - it's what the API itself reads from.

The CLI sync command should pull from GitHub releases or the npm @iconify-json/{prefix} packages, NOT

from the live HTTP API. This gives: reproducible builds, offline-capable syncing, no API load, version pinning.

Concrete sync strategy for the CLI:

- Primary: fetch individual collection JSON files from the iconify/icon-sets GitHub repo via GitHub releases or raw URLs
- Secondary: support @iconify-json/{prefix} npm package structure as an alternative input (parse the icons.json file format)
- Never: call api.iconify.design/collection?prefix=mdi for bulk data sync
- Runtime remote fallback (in app): MAY call the HTTP API, but only when explicitly opted-in and only in debug mode by default

## **2.4 - Icon License Complexity Is a First-Class Problem**

The original plan mentions licensing but treats it as a checkbox. Research shows it's more complex:

- 208 icon sets with 208 different licenses - MIT, Apache 2.0, OFL, CC BY, CC BY-SA, custom
- Some sets have attribution requirements that are legally binding for commercial apps
- Iconify's own framework is MIT but explicitly states 'this license does not apply to icons'
- A Flutter developer who unknowingly ships an attribution-required icon set without attribution in their app is in legal violation

REQUIRED ADDITIONS:

- CLI doctor must warn on attribution-required collections by default
- Generated license manifest must clearly separate: MIT/permissive sets vs attribution-required sets
- Documentation must include a 'safe collections' list (fully MIT/Apache permissive, no attribution needed)
- iconify.yaml should support a license_policy: strict flag that fails the build if attribution-required sets are included without manifest

## **2.5 - The 4-Week Timeline Is Not Realistic**

The original plan's Week 1-4 roadmap to 'release candidate' is fiction for a solo developer.

Based on the actual scope: 3 packages, CLI with 6+ commands, golden tests, docs site, sync pipeline.

Realistic estimate for a quality v1: 12-16 weeks of focused part-time work, or 10-12 weeks full-time.

Shipping prematurely will produce exactly the kind of abandoned package this project aims to replace.

See Section 9 for the revised realistic timeline.

## **2.6 - build_runner vs CLI Was Never Properly Evaluated**

The plan chose CLI-only without evaluating the build_runner alternative. This matters because Flutter devs already run build_runner for json_serializable, freezed, and Riverpod. Adding another CLI command breaks that mental model.

| **Approach**         | **Pros**                                                                                                                            | **Cons**                                                                                                       | **Verdict**                   |
| -------------------- | ----------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- | ----------------------------- |
| build_runner builder | Integrates with existing workflow Watch mode works automatically Familiar to Flutter devs                                           | Slow on large projects Harder to expose CLI-style commands (sync, doctor, diff) Complex to implement correctly | Good for codegen only         |
| Dart CLI (dart run)  | Full control over commands Faster for bulk operations Easier to implement search/diff/doctor Explicit is better for sync operations | Extra step outside build_runner Devs must remember to run it                                                   | Best for sync/doctor/licenses |
| Hybrid (both)        | Codegen via build_runner Sync/doctor/diff via CLI Best of both worlds                                                               | Two things to maintain Potentially confusing                                                                   | RECOMMENDED                   |

DECISION: Adopt the hybrid approach. The build_runner builder handles code generation (watches files, regenerates icons.g.dart on change). The CLI handles sync, doctor, diff, licenses, and init. These are complementary, not redundant.

## **2.7 - Starter Registry Needs a Concrete Definition**

The plan says 'ship a small starter registry' but never specifies what's in it. This needs to be locked before implementation.

Proposed starter registry contents:

| **Collection**        | **Prefix** | **Icon Count to Bundle** | **License** | **Reason**                   |
| --------------------- | ---------- | ------------------------ | ----------- | ---------------------------- |
| Material Design Icons | mdi        | Top 150 by usage         | MIT         | Most popular, universal      |
| Lucide                | lucide     | Top 100 by usage         | ISC         | Clean, modern, growing fast  |
| Tabler Icons          | tabler     | Top 100 by usage         | MIT         | Developer-favorite           |
| Heroicons             | heroicons  | All (~300)               | MIT         | Tailwind community crossover |
| Collection metadata   | all 208    | Names + license only     | -           | Enables smart error messages |

Total starter bundle target: under 200KB uncompressed. This is achievable. MDI full set is ~3MB; top-150 icon bodies alone are ~50KB.

# **4\. Revised Out-of-Box Strategy**

The second document's Hybrid Auto Mode concept is correct in spirit but wrong in execution. This section is the definitive version.

## **4.1 - The Corrected Default**

| **Build Mode** | **Default Behavior**                                                     | **Rationale**                                |
| -------------- | ------------------------------------------------------------------------ | -------------------------------------------- |
| debug          | Starter registry → memory cache → remote fetch → fallback                | Dev convenience, no production impact        |
| profile        | Starter registry → memory cache → remote fetch → fallback                | Same as debug for testing                    |
| release        | Starter registry → generated subset → persistent cache → fallback widget | No external calls unless explicitly opted in |

This is the correct position: development is frictionless, production is honest about dependencies.

## **4.2 - The Four Modes (Unchanged from v2 doc, now correct)**

- **auto - default (with corrected release behavior above)**
- offline - strict local only, best for regulated/deterministic builds
- generated - strict generated subset mode
- remoteAllowed - explicit opt-in for production remote fetch (enterprise, internal tools)

## **4.3 - The Developer Journey (Revised)**

- Install package. Use IconifyIcon('mdi:home'). Works in debug via starter registry + remote fallback.
- Run dart run iconify_flutter_cli init. Creates iconify.yaml.
- Run dart run iconify_flutter_cli sync --collections mdi,lucide. Downloads data locally.
- Add iconify_flutter_builder to dev_dependencies. Run dart run build_runner build. Generates icons.g.dart.
- Switch to generated mode in release config. App is now fully offline, deterministic, no external calls.

The transition from Step 1 to Step 5 must not require changing any call site.

IconifyIcon('mdi:home') in Step 1 and Step 5 must behave identically from the widget API perspective.

This is the non-negotiable DX constraint that everything else is built around.

# **5\. Revised Data Pipeline**

## **5.1 - Upstream Sources (Corrected)**

Primary data source for CLI sync: iconify/icon-sets GitHub repository

- URL pattern: <https://raw.githubusercontent.com/iconify/icon-sets/master/json/{prefix}.json>
- This is the same data the HTTP API serves, without hitting the API
- Version-pinnable by referencing specific git commits or GitHub releases
- Updated 3x/week automatically by Iconify

Secondary: direct npm package structure compatibility

- Parse @iconify-json/{prefix}/icons.json format for teams who have it locally
- Useful for offline enterprise environments that mirror npm

Runtime remote fallback (app only, debug mode only by default):

- <https://api.iconify.design/{prefix}.json?icons={name1},{name2}> - targeted fetch
- Must include User-Agent header identifying the package version
- Must batch requests: fetch multiple icons in single request where possible

## **5.2 - The iconify.json Format (What You're Parsing)**

The canonical format is well-documented. Key fields to understand:

{ "prefix": "mdi",

"icons": {

"home": { "body": "&lt;path d=\\'...\\'/&gt;", "width": 24, "height": 24 }

},

"aliases": { "home-outline": { "parent": "home-outline-alt" } },

"width": 24, "height": 24,

"info": { "name": "Material Design Icons", "author": {...}, "license": {...} }

}

Critical notes:

- body field contains SVG path content WITHOUT the outer &lt;svg&gt; tag - you must wrap it
- width/height at collection level are defaults; individual icons can override
- aliases can be chains - alias points to parent which may itself be an alias; must resolve recursively
- currentColor is pre-normalized by Iconify Tools for monotone icons - this is why color theming works

# **6\. Rendering Strategy (New Section)**

This section was entirely missing from v1. Given the Impeller issues, rendering strategy needs explicit treatment.

## **6.1 - Rendering Modes**

| **Mode**     | **Implementation**                                                    | **When to Use**            | **Impeller Safe?**      |
| ------------ | --------------------------------------------------------------------- | -------------------------- | ----------------------- |
| svgWidget    | flutter_svg SvgPicture.string()                                       | Default, no color override | Mostly yes              |
| svgWithColor | SvgStringLoader with SvgTheme(currentColor: color)                    | With color override        | NO - broken on Impeller |
| rasterized   | Render SVG to ui.Image at target DPR, cache as ImageProvider          | Color override + Impeller  | Yes                     |
| auto         | Detects Impeller, falls back to rasterized for color-overridden icons | Default for color: param   | Yes                     |

Implementation note for the rasterized path:

- Use vg.loadPicture() from flutter_svg internals
- Render to ui.Image at devicePixelRatio \* target size
- Cache keyed by: iconName + color + size + DPR
- Do NOT rasterize on the main isolate for large icon grids - use compute()

## **6.2 - currentColor Normalization**

Iconify Tools pre-processes all monotone icons to use currentColor. This means color theming works by wrapping the SVG body with a theme: parameter. Your package must:

- Detect whether an icon uses currentColor (check body string for 'currentColor')
- If yes: use SvgTheme(currentColor: color) approach
- If no (multicolor icon): do not apply color override by default; document this clearly
- Expose a forceColor: parameter for advanced override with explicit warning in docs

# **8\. Correctness Plan (Additions to v1)**

## **8.1 - Impeller-Specific Golden Tests**

The v1 plan had golden tests but missed the Impeller dimension entirely.

- Must run golden tests on: Skia renderer AND Impeller renderer, separately
- Target real devices, not just emulators (emulator Impeller behavior differs)
- Cover: monochrome icon + color override, multicolor icon, stroke icon with color
- Add to CI matrix: physical iOS device (Impeller on by default) + Android with --enable-impeller

## **8.2 - Alias Chain Resolution Tests**

The iconify JSON format allows aliases of aliases. The v1 plan missed this edge case.

- Test: direct icon lookup
- Test: alias → icon
- Test: alias → alias → icon (chain depth 2)
- Test: circular alias detection (should not infinite loop)
- Test: alias with property overrides (width/height can differ from parent)

## **8.3 - Flutter Web Rendering**

The v1 plan listed web in the platform matrix but said nothing specific about web rendering.

- flutter_svg on web uses a different rendering path than native
- Test: icons render correctly in CanvasKit renderer AND HTML renderer
- Known issue: some SVG features (filters, masks) behave differently on web HTML renderer
- Documentation must note which icon features are limited on Flutter web HTML renderer

# **10\. What Is Still Missing From Both Documents**

## **10.1 - Custom Icon Set Support**

Neither document addresses enterprise users with proprietary icons. This is a real need:

- Teams with brand icons that are not in any public set
- Teams with icon variations (outlined vs filled variants not in upstream)

REQUIRED ADDITION: iconify.yaml should support a custom_sets: block pointing to local JSON files in the same IconifyJSON format. The CLI sync should be able to ingest them. The runtime providers should serve them.

## **10.2 - VS Code / IntelliJ Extension**

The plan mentions IDE autocomplete via generated constants but misses the bigger opportunity: an IDE extension that shows icon previews inline when hovering over an icon name string. This exists for web (Iconify IntelliSense for VS Code). A Flutter equivalent would be a significant DX differentiator. This is post-v1 but should be in the roadmap.

## **10.3 - Snapshot Storage Strategy**

The plan says 'pin snapshots' but never says where they are stored. Concrete answer needed:

- Pinned snapshots are stored in the user's repo under data/snapshots/ or a configurable path
- They are committed to version control - this is intentional and correct for reproducibility
- Size concern: a single collection JSON (e.g., mdi) is ~3MB. Storing 5 collections = ~15MB committed. This is acceptable. Document the tradeoff.
- gitignore: generated code (icons.g.dart) should be gitignored by default; snapshots should NOT be

## **10.4 - The 'Safe Collections' Guide for Developers**

Most Flutter developers will not read license metadata. They will install the package and use whatever collection looks good. The plan needs a curated 'safe for commercial use, no attribution required' list in the documentation. Based on research:

- MIT: mdi, tabler, heroicons, bootstrap-icons, carbon, ant-design, ph (Phosphor Icons)
- ISC: lucide
- Apache 2.0: material-symbols, ic (Google Icons)
- Requires attribution: simple-icons (CC0 but with attribution guidelines), many emoji sets
- Do NOT bundle by default without warning: any CC BY-SA or GPL licensed sets

## **10.5 - Platform-Specific Edge Cases**

The plan lists all platforms but doesn't address platform-specific issues:

- iOS: Impeller on by default since Flutter 3.10. Color filter bug is production-affecting today.
- Android: Impeller opt-in but becoming default. Same bug applies.
- Flutter Web (HTML renderer): Some SVG attributes unsupported. Limited to CanvasKit for full fidelity.
- Flutter Web (CanvasKit): Full SVG support but larger bundle size. Icons add to this.
- macOS/Linux/Windows desktop: No known special issues. Test basic rendering.
- WASM: Flutter WASM is emerging. SVG rendering via flutter_svg has not been tested widely. Earmark as 'experimental' in v1.

# **14\. Final Verdict**

The original plan is architecturally sound. The product thesis is correct. The canonical identity model,

the three-package structure, the CLI design, the governance thinking - all of that was right.

What was missing: seven specific technical and ethical details that would have caused real production

problems if not caught before implementation began. Those are now addressed in this document.

The correct build order is: core engine → Flutter package (with Impeller fix) → build_runner builder → CLI.

Do not invert this order. The CLI is the most complex piece and benefits from having the other three

working first.

Do not ship until the Impeller color rendering path is validated on a real physical device.

That is the single highest-risk technical item in the entire project.
